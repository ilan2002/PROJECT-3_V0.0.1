**Project:** Project-3  
**Date:** 2025-02-14  
**Version:** 1.0

## 1. Introduction & Purpose
This Software Design Document (SDD) defines the architecture and detailed design for Project-3. It follows IEEE 1016 and specifies structural decomposition, component responsibilities, interfaces, data design, key design decisions, and component-to-requirement traceability. The design targets an embedded control system with safety monitoring, UART command interfaces, configuration management, and artifact generation for release.

---

## 2. System Architecture Overview
### 2.1 Architectural Style
The system employs a **layered architecture** with the following tiers:
- **Application/Control Layer:** state machines, control algorithms, safety gates.
- **Service Layer:** schedulers, validation, diagnostics, command processing.
- **Middleware/Driver/HAL:** hardware access, timers, ADC, UART, watchdog.
- **Persistence & Configuration:** configuration storage, NVM repositories, log stores.
- **Release/Tooling:** compliance and artifact generation.

### 2.2 Key Runtime Flows
- **Control Loop Execution:** Periodic scheduler triggers ADC sampling → validation/scaling → control computation → actuator drive.
- **Command Handling:** UART parsing → validation → state transitions/actuator enable/disable → acknowledgements.
- **Safety Monitoring:** Continuous ADC validity, range checks, over/under voltage, overcurrent, thermal, stall, saturation → fault handling and safe state.
- **Startup Safety:** hardware outputs disabled until initialization and delay conditions are met.

### 2.3 Architectural Dependencies (High-Level)
- Control logic depends on ADC acquisition and configuration services.
- Safety modules depend on fault manager and actuator driver.
- UART services depend on driver, parser, and command routing.
- State management relies on fault/event buses and recovery handlers.

## 3. Module Descriptions
Each module definition includes purpose, components, and interactions.

### 3.1 ControlModule
**Purpose:** Execute control algorithms and actuator command generation.  
**Components:** DD-COMP-001, 002, 016, 020, 027, 030, 031, 037, 052, 088, 089, 092, 110, 112, 115.  
**Interactions:** Reads ADC samples from Sensor/Sensing modules, uses ConfigModule parameters, outputs to Actuator/Driver/HAL.

### 3.2 SensorModule
**Purpose:** Sample and provide ADC feedback.  
**Components:** DD-COMP-003, 111.  
**Interactions:** Uses ADC drivers and publishes sample data to safety/control logic.

### 3.3 CommModule / UartModule / CommsModule / CommunicationModule / UartCommModule
**Purpose:** UART command parsing, diagnostics, acknowledgements, and status reporting.  
**Components:** DD-COMP-004, 005, 006, 007, 028, 029, 032, 058, 059, 060, 061, 063, 093, 094, 095, 096, 097, 098, 114.  
**Interactions:** Interfaces with UART driver, command handlers, state machine, and diagnostics.

### 3.4 ConfigModule
**Purpose:** Provide runtime configuration and NVM-backed settings.  
**Components:** DD-COMP-008, 009, 025, 026, 043, 049, 065, 066.  
**Interactions:** Supplies control gains, thresholds, voltage limits to control and safety modules.

### 3.5 SafetyModule / SafetyControlModule / FaultMonitoringModule / PowerSafetyModule / PowerMonitoringModule
**Purpose:** Detect invalid conditions, enforce safe state, and report faults.  
**Components:** DD-COMP-010, 011, 012, 013, 014, 022, 023, 038, 040, 042, 044, 045, 050, 051, 053, 055, 056, 057, 090, 091, 099, 100, 101, 102, 103, 105, 113.  
**Interactions:** Uses ADC data, config thresholds, and actuates safe/disable commands via actuator interfaces.

### 3.6 SchedulerModule / ControlLoopModule / ADCModule / AdcModule
**Purpose:** Deterministic scheduling for control and ADC sampling.  
**Components:** DD-COMP-021, 033, 034, 035, 069, 070, 071, 072, 089, 110.  
**Interactions:** Uses timers and drivers to trigger ADC capture and control execution.

### 3.7 SignalProcessingModule / SensorProcessingModule
**Purpose:** Scale and process raw sensor data.  
**Components:** DD-COMP-019, 106.  
**Interactions:** Uses calibration repositories and provides normalized outputs.

### 3.8 HAL / HALModule / HardwareAbstractionLayer / Driver / Middleware
**Purpose:** Provide hardware interface implementations.  
**Components:** DD-COMP-017, 018, 024, 046, 076, 077, 078, 079, 080.  
**Interactions:** Hardware drivers for ADC, UART, PWM, GPIO, timers.

### 3.9 StartupModule
**Purpose:** Enforce post-reset safety delays.  
**Components:** DD-COMP-067, 068.  
**Interactions:** Coordinates with timers and control loop scheduler to enable outputs.

### 3.10 Application / StateModule / StateManagementModule / StateMachineModule
**Purpose:** System state transitions and fault-to-state handling.  
**Components:** DD-COMP-081, 082, 083, 084, 120, 121, 122, 123, 124, 125.  
**Interactions:** Consumes fault events and commands; updates system state.

### 3.11 Logging & Data Layers
**Purpose:** Persist fault logs and invalid command logs.  
**Components:** DD-COMP-064, 118, 119, 107.  
**Interactions:** Time services and NVM storage.

### 3.12 Compliance & Release Tooling
**Purpose:** MISRA compliance and artifact generation.  
**Components:** DD-COMP-015, 126, 127, 128, 129, 130, 131, 132, 133, 134.  
**Interactions:** Documentation repositories, EDA tools, packaging.

## 5. Component Design Details (Key Behavioral Notes)
- **Deterministic Scheduling:** Control loops at 1 ms or 10 ms depending on subsystem requirement (DD-COMP-021, 089, 110).
- **Safety Interlocks:** Guard and inhibit layers enforce raw ADC bounds, freshness, and fault state gating (DD-COMP-038, 040, 042, 044, 051, 052).
- **Fault Handling:** Multi-source fault aggregation, deterministic transitions, retention logic (DD-COMP-012, 053, 055, 084, 105, 121).
- **UART Latency Contracts:** Parsing and acknowledgements ensure ≤50 ms for command dispatch and ≤20 ms for ACK (DD-COMP-028, 029, 032).
- **Raw ADC Policy:** Control computations strictly use raw ADC data; conversion to engineering units is segregated (DD-COMP-092, 088, 106).

## 6. Data Design
### 6.1 Key Data Models
- **ADC Sample Buffer:** `{ value:int, timestamp_us:uint64 }` (DD-COMP-036).
- **Fault Event:** `{ fault_code, timestamp, severity }` (DD-COMP-116/118).
- **System State Status:** `{ current_state, last_transition_cause }` (DD-COMP-124).
- **Configuration Sets:** 
  - Control gains (Kp/Ki/Kd), ADC sampling rates, UART timeouts (DD-COMP-008/009).
  - Shipping threshold (DD-COMP-025/026).
  - Overvoltage threshold (DD-COMP-049).
  - Voltage range (DD-COMP-066).

### 6.2 Database/Store Concepts
- **ConfigStore (NVM/RAM):** key/value with defaults.
- **CalibrationStore:** min/max ADC values for scaling.
- **FaultLogStore:** append-only fault records with timestamp.

### 6.3 Persistence
- **NvmConfigStorage:** integrity checks on format/version only (DD-COMP-107).

---

## 7. Interface Design
### 7.1 UART Interfaces
- **Command Frames:** ON/OFF commands, diagnostic requests, ACK responses.
- **Baud/Format:** 115200, 8N1 enforced (DD-COMP-007, 095, 096, 097).
- **Latency Constraints:** ACK within 20 ms, command dispatch within 50 ms.

### 7.2 Internal APIs
- **StateStatusAPI:** `GET /state` → `{state, cause}` (DD-COMP-125).
- **Hardware Abstraction Interface:** `read_adc()`, `write_uart()`, `set_gpio()` (DD-COMP-080).

### 7.3 Timing Interfaces
- **Scheduler Hooks:** `on_tick()` for loop tasks (DD-COMP-021, 089, 110).
- **Watchdog:** `watchdog_kick()` (DD-COMP-078, 086, 104).

---

## 8. Design Decisions & Rationale
1. **Observer Pattern for Events:** Ensures timely propagation for faults and state changes without tight coupling.
2. **State Machines for Safety and Control:** Deterministic transitions required for safety compliance.
3. **Raw ADC-Only Policy:** Ensures control path uses unmodified sensor data; reduces conversion-induced errors.
4. **Layered Architecture Enforcement:** Prevents direct hardware access from application logic.
5. **NVM Repository Pattern:** Simplifies persistence and version management.
6. **Deterministic Scheduling:** Fixed-period loops to meet control and safety timing requirements.
7. **Safety Guards & Inhibitors:** Multiple gates enforce independent safety constraints.
